# ADR-001: Выбор стека для MVP realtime-квиза

## Статус

Accepted

## Контекст

MVP должен быстро проверить продуктовый сценарий, близкий к Kahoot: организатор
создает квиз, запускает live-сессию, участники подключаются по коду и отвечают
почти синхронно. Критерии выбора: realtime-взаимодействие, стабильность,
скорость разработки MVP и понятный путь к следующей итерации.

## Решение

Используем:

- Next.js App Router + React + TypeScript для веб-интерфейса.
- Tailwind CSS для быстрой сборки интерфейса без отдельной дизайн-системы.
- Кастомный Node.js `server.ts`, который поднимает Next.js и Socket.IO в одном
  HTTP-процессе и на одном порту.
- Socket.IO для realtime-событий с fallback-транспортами и комнатами в будущей
  игровой логике.
- PostgreSQL как основное реляционное хранилище.
- Prisma как typed ORM, миграционный инструмент и единый контракт схемы.
- JWT в httpOnly-cookie для будущей авторизации с ролями `ORGANIZER` и
  `PARTICIPANT`.

## Обоснование

### Realtime

Socket.IO закрывает базовую механику live-квиза: подключения участников,
события вопросов/ответов, lobby-комнаты, disconnect/reconnect и broadcast на
сцену или конкретную сессию. Интеграция в кастомный сервер убирает отдельный
realtime-сервис на этапе MVP и снижает операционную сложность.

### Стабильность

PostgreSQL хорошо подходит для транзакционных данных: пользователи, квизы,
сессии, ответы, начисленные баллы и история прохождений. Prisma дает миграции,
строгие типы и явные связи, что уменьшает риск расхождения кода и схемы БД.

### Скорость MVP

Next.js App Router позволяет держать страницы, серверные обработчики будущего
API и клиентские компоненты в одном приложении. TypeScript и Prisma ускоряют
изменение модели без потери проверяемости. Tailwind снижает стоимость верстки
первых экранов, не блокируя последующее выделение дизайн-системы.

## Последствия

- В MVP один Node.js-процесс обслуживает и HTTP, и WebSocket-трафик.
- При росте нагрузки Socket.IO потребует адаптера для горизонтального
  масштабирования, например Redis adapter.
- JWT/httpOnly-cookie оставляет авторизацию простой для браузерного клиента, но
  потребует аккуратной CSRF-стратегии для state-changing HTTP-запросов.
- Prisma-схема становится основным источником правды по данным и должна
  изменяться только миграциями.
