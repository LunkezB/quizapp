# QuizApp

MVP realtime-квиза: Next.js App Router, React, TypeScript, Tailwind,
кастомный Node.js-сервер с Socket.IO в одном процессе, PostgreSQL и Prisma.

Реализовано: регистрация/логин (JWT в httpOnly cookie), CRUD квизов и
вопросов с вариантами ответа, переупорядочивание вопросов, проверка владения
ресурсом на сервере, живая игровая сессия по WebSocket (лобби → вопросы с
таймером → подсчёт очков → финальный лидерборд) и личные кабинеты с историей
прошлых игр для участника и организатора. Сервер — единственный источник
истины: клиент не считает время, не определяет правильность ответа и не
начисляет баллы.

## Требования

- Node.js 22+
- npm 10+
- Docker и Docker Compose

## Локальный запуск

1. Установить зависимости:

```bash
npm install
```

2. Поднять PostgreSQL:

```bash
docker compose up -d
```

3. Создать локальный `.env` из примера и при необходимости поменять секреты.
   По умолчанию PostgreSQL публикуется на локальном порту `5433`.

```bash
cp .env.example .env
```

4. Применить миграции:

```bash
npm run prisma:migrate
```

5. Засеять категории квизов (Общие знания, История, Наука, Кино, Музыка,
   Спорт, География, IT и т.д.):

```bash
npm run prisma:seed
```

6. Запустить Next.js + Socket.IO на одном порту:

```bash
npm run dev
```

По умолчанию приложение доступно на `http://localhost:3000`.

На стартовой странице есть Socket.IO health check: клиент отправляет событие
`ping`, сервер отвечает событием и callback-ответом `pong`.

## Auth и CRUD

- `/register`, `/login` — регистрация и вход. Пароли хешируются (bcrypt),
  сессия — JWT в httpOnly+SameSite cookie (см. `src/lib/auth-token.ts`).
- `/dashboard` — список своих квизов и форма создания.
- `/quiz/[id]/edit` — редактор метаданных квиза и списка вопросов
  (добавление, редактирование, удаление, переупорядочивание).
- Роль (`ORGANIZER`/`PARTICIPANT`) — только стартовый сценарий; управлять
  своими квизами может любой залогиненный пользователь. Владение проверяется
  на сервере по `ownerId` в каждом Server Action.
- Защищённые маршруты (`/dashboard`, `/quiz/*`, `/join`, `/host/*`, `/play/*`,
  `/history/*`, `/host-history/*`) требуют валидный JWT-cookie. Проверка
  продублирована на двух уровнях: `middleware.ts` (для случаев без
  кастомного сервера) и в `server.ts` напрямую — Next.js middleware не
  гарантированно вызывается при использовании кастомного HTTP-сервера вместе
  с Turbopack, поэтому `server.ts` — основной уровень защиты в dev/prod.

## Игровая сессия в реальном времени

Полный протокол socket-событий и payload'ов описан в `docs/realtime-protocol.md`
(локальный файл для разработки, не в git — см. раздел «Структура»). Как сыграть:

1. Организатор открывает свой квиз (`/quiz/[id]/edit`) и жмёт **Запустить**
   (кнопка активна только если в квизе есть хотя бы один вопрос). Открывается
   `/host/[code]` с крупным 6-символьным кодом комнаты (без похожих символов
   `O`/`0`/`I`/`1`).
2. Каждый участник открывает **Подключиться** в шапке (`/join`),
   вводит код и никнейм для этой игры → попадает в `/play/[code]`. Участники
   появляются в лобби хоста мгновенно.
3. Когда в лобби есть хотя бы один участник, хост жмёт **Start** — начинается
   первый вопрос с серверным дедлайном; у участников кнопки-варианты и
   обратный отсчёт, у хоста — вид "сцены" со счётчиком ответивших.
4. По закрытии вопроса (дедлайн или все ответили) хост видит разбивку
   ответов и промежуточный лидерборд, жмёт **Следующий вопрос**/**Завершить
   игру**. После последнего вопроса всем показывается финальный лидерборд.
5. Участник может обновить страницу в любой момент — он переподключается по
   коду и восстанавливает свой счёт (сопоставление по пользователю, а не по
   сокету). Дисконнект хоста не роняет игру: участники видят баннер паузы,
   таймер вопроса продолжает идти на сервере, хост восстанавливает
   управление при повторном заходе на `/host/[code]`.

Игровое состояние во время сессии живёт в памяти процесса (`Map<code,
RoomState>` в `src/server/game-store.ts`, общий и для Server Actions, и для
socket-обработчиков через `globalThis`); в Postgres пишутся жизненный цикл
сессии и все ответы участников — на этих данных построена история игр ниже.

## Продакшн-деплой

Полный пошаговый runbook — в [`deploy/DEPLOY.md`](deploy/DEPLOY.md):
Docker Compose (`docker-compose.prod.yml` + многоступенчатый `Dockerfile`),
reverse-proxy nginx с проксированием WebSocket (`deploy/nginx/`), выпуск
TLS-сертификата через certbot, обновление версии, откат, альтернатива без
Docker через systemd (`deploy/systemd/quizapp.service`).

- `GET /api/health` — реализован прямо в `server.ts`, отдаёт `200
  {"status":"ok","db":"ok",...}`, если процесс жив и Postgres отвечает,
  иначе `503`. Используется Docker `HEALTHCHECK` и `docker compose`.
- Продакшн-переменные окружения — шаблон в `.env.production.example`
  (копируется в `.env.production`, не коммитится).
- В проде, как и в dev, работает единственный процесс `server.ts`:
  serverless/Vercel не подходит — WebSocket-соединениям нужен постоянный
  процесс, а игровое состояние живёт в памяти и не переживает рестарт
  контейнера (подробнее — в `deploy/DEPLOY.md`).

## История игр (личный кабинет)

Только read-запросы поверх уже записанных `Session`/`SessionParticipant`/
`ParticipantAnswer` — игровая логика не затрагивается.

- **Участник**: `/history` — список завершённых игр, в которых пользователь
  участвовал (место, очки, дата, число участников). `/history/[sessionId]`
  — детализация: что выбрал пользователь на каждом вопросе, правильно ли,
  сколько очков. Доступно только тому, кто реально участвовал в этой сессии
  (проверка `SessionParticipant.userId` на сервере) — иначе `not-found`.
- **Организатор**: `/host-history` — список сессий, где пользователь был
  хостом: активные (LOBBY/IN_PROGRESS, со ссылкой на `/host/[code]` для
  возврата в игру) отдельно от завершённых (с победителем).
  `/host-history/[sessionId]` — финальный лидерборд и статистика по каждому
  вопросу (доля правильных ответов, распределение по вариантам). Доступно
  только `Session.hostId` этой сессии — иначе `not-found`.
- `/dashboard` дополнительно показывает по 5 последних записей из «Я
  проводил» и «Я играл» со ссылками на полные списки.
- Место (`rank`) участника считается сортировкой `totalScore` по убыванию с
  тем же порядком, что и во время игры на живом лидерборде (см.
  `rankParticipants` в `src/lib/history-queries.ts`), чтобы историческое
  место всегда совпадало с тем, что участник видел в конце игры.

## Структура

- `server.ts` - кастомный HTTP-сервер Next.js с Socket.IO, дублированной
  проверкой auth-cookie для защищённых маршрутов и регистрацией игровых
  socket-обработчиков (`registerGameHandlers`).
- `middleware.ts` - та же проверка auth на уровне Next.js middleware.
- `src/app` - App Router: лендинг, auth, dashboard, редакторы квиза/вопроса,
  `/join`, `/host/[code]`, `/play/[code]`, `/history[/[sessionId]]`,
  `/host-history[/[sessionId]]`.
- `src/actions` - Server Actions для auth, CRUD квизов/вопросов и
  `startSessionAction` (создание игровой сессии) (zod-валидация).
- `src/lib/auth.ts`, `src/lib/auth-token.ts` - JWT-сессия, `getCurrentUser()`/`requireCurrentUser()`.
- `src/lib/realtime.ts` - общий контракт socket-событий: имена событий,
  zod-схемы входящих payload'ов, TS-типы исходящих payload'ов.
- `src/lib/history-queries.ts` - агрегирующие read-запросы для истории игр
  (список участий, список проведённых сессий, ранжирование лидерборда) —
  без N+1, переиспользуются `/history`, `/host-history` и `/dashboard`.
- `src/server/game-store.ts` - in-memory реестр комнат (`Map<code,
  RoomState>` на `globalThis`), формула баллов, построение публичного
  снимка состояния без `isCorrect`.
- `src/server/game-socket.ts` - обработчики socket-событий игровой сессии
  (лобби, вопрос/таймер, reveal/лидерборд, финал, персист в БД).
- `src/components/host-game-client.tsx`, `play-game-client.tsx` - клиентские
  компоненты экрана хоста/участника (сокет-подключение, рендер состояния).
- `src/components/socket-status.tsx` - клиентская проверка socket ping/pong.
- `src/components/ui/` - переиспользуемые presentational-компоненты
  (Button, Card, Badge, Field, Reveal) для единого визуального языка.
- `prisma/schema.prisma` - доменная модель MVP.
- `prisma/seed.ts` - сид категорий квизов.
- `docker-compose.yml` - локальный PostgreSQL для разработки.
- `Dockerfile`, `docker-compose.prod.yml` - продакшн-сборка приложения и
  стек (app + Postgres) для деплоя.
- `deploy/` - runbook (`DEPLOY.md`), entrypoint-скрипт с миграциями,
  конфиг nginx (reverse proxy + WebSocket), systemd-юнит как альтернатива
  Docker.
- `docs/` - ADR, протокол socket-событий и UX-спецификация; хранятся
  локально для разработки, в git не отслеживаются (см. `.gitignore`).
